# 多角度评审框架

> 一道题做得好不好，不能只有自己说了算。
> 需要从多个角色的视角来审视，每个角色关注的重点不同。
> 只有所有角度都认可，才是真正有水准的作品。

---

## 角色一：评委（决定你的分数）

### 评委关注什么？
1. **基本要求是否全部完成？** 这是底线，缺一项就可能掉档
2. **指标是否达标？** 不是"差不多"，而是精确到题目要求的数字
3. **发挥部分完成了几项？** 区分一等奖和二等奖的关键
4. **报告质量如何？** 方案比较有没有？理论推导对不对？测试数据完整吗？
5. **作品的工程规范性** 测试点标识清晰吗？接线整洁吗？操作方便吗？

### 评委的扣分点（血泪教训）
- 报告没有摘要 → 扣2~3分
- 没有方案比较（只写了一个方案）→ 扣2~4分
- 测试数据不完整（只测了一个点）→ 扣3~5分
- 图表没有编号和标题 → 扣1~2分
- 作品操作不方便（评委要自己接线）→ 印象分差
- 一键启动后需要人工干预 → 直接扣分

### 评委视角的自检清单
- [ ] 基本要求每一项都有对应的测试数据
- [ ] 每项指标都标注了"达标/未达标"
- [ ] 报告结构完整（摘要→方案→理论→电路→测试→总结）
- [ ] 所有图表有编号
- [ ] 测试点有标识，评委能方便地测量

---

## 角色二：资深工程师（评判你的技术深度）

### 工程师关注什么？
1. **方案选择是否合理？** 不是最复杂的方案最好，而是最适合的
2. **关键技术决策有没有依据？** 为什么选这个MCU？为什么用这个算法？
3. **误差分析是否到位？** 每个误差源都识别了吗？量化了吗？
4. **代码质量如何？** 模块化、可读性、鲁棒性
5. **有没有考虑边界情况？** 最大值、最小值、异常输入

### 工程师会问的尖锐问题
- "你的ADC采样率为什么选这个值？奈奎斯特够吗？有过采样吗？"
- "你的FFT窗函数为什么选平顶窗而不是汉宁窗？"
- "你的PID参数是怎么整定的？有理论依据还是纯试凑？"
- "如果输入信号频率不在你的测量范围内，系统会怎样？"
- "你的系统延迟是多少？对结果有什么影响？"

### 工程师视角的自检清单
- [ ] 每个技术决策都有"为什么选"和"为什么不选其他"
- [ ] 误差预算表：列出每个误差源及其量化值
- [ ] 代码有清晰的模块划分和接口定义
- [ ] 边界情况有处理（输入为零、输入超范围、信号丢失等）
- [ ] 性能指标有理论预测值和实测值的对比

---

## 角色三：带队老师（评判你的备赛价值）

### 老师关注什么？
1. **这份资料能不能直接用于教学？** 学生看了能不能学到东西？
2. **技术路线是否可复现？** 另一个队伍照着做能不能做出来？
3. **时间分配是否合理？** 4天3夜够不够？哪里可能卡住？
4. **物料是否容易获取？** 有没有用到稀缺或昂贵的器件？
5. **对未来出题有没有预判价值？** 这道题的技术能不能迁移到其他题目？

### 老师视角的自检清单
- [ ] 从零开始到完成，步骤是否清晰可执行
- [ ] 物料清单是否完整，能不能一次性采购齐
- [ ] 调试顺序是否合理（先易后难，逐步集成）
- [ ] 有没有标注"容易卡住的地方"和"解决方案"
- [ ] 技术点能不能迁移到其他题目

---

## 角色四：参赛学生（评判你的实用性）

### 学生关注什么？
1. **我能不能看懂？** 代码注释够不够？原理解释清不清楚？
2. **我能不能照着做？** 接线图有没有？CubeMX怎么配置？
3. **出了问题我能不能排查？** 故障排查表有没有？
4. **时间够不够？** 4天3夜能不能做完？
5. **我的基础够不够？** 需要什么前置知识？

### 学生视角的自检清单
- [ ] 代码每个函数都有注释说明功能和参数
- [ ] 有完整的引脚定义表（不需要翻代码找引脚）
- [ ] 有CubeMX配置说明（不需要猜外设怎么配）
- [ ] 有调试步骤（先调什么后调什么）
- [ ] 有"如果XXX不工作怎么办"的故障排查

---

## 角色五：产品经理/系统架构师（评判你的系统思维）

### 他们关注什么？
1. **系统架构是否清晰？** 模块之间的接口定义明确吗？
2. **有没有单点故障？** 某个模块挂了整个系统会怎样？
3. **可扩展性如何？** 如果需求变了（比如频率范围扩大），改动大不大？
4. **资源利用是否合理？** CPU占用率、内存使用、功耗
5. **用户体验如何？** 操作是否直观？显示信息是否有用？

### 系统视角的自检清单
- [ ] 有系统架构图（模块+接口+数据流）
- [ ] 模块之间低耦合（改一个模块不影响其他）
- [ ] 有状态机图（系统在什么状态下做什么）
- [ ] 资源使用有预算（CPU时间、RAM、Flash）
- [ ] 异常处理有考虑（过压保护、看门狗、错误恢复）

---

## 如何使用这个框架

每道题做完后，用这5个角色的自检清单逐一审查。
如果某个角色的某项不通过，就回去改，直到全部通过。

**终极标准**：
- 评委给满分
- 工程师说"这个方案我认可"
- 老师说"这份资料我可以直接用"
- 学生说"我照着做能做出来"
- 架构师说"这个系统设计合理"

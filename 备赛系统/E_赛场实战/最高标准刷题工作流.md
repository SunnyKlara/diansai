# 最高标准刷题工作流

> 这套流程从 2024 H、2025 G 两道题实战中沉淀出来，目标：**4~6 小时内交付一份对得起国一的完整工程**。
> 配套：`.kiro/specs/_template/`（spec 蓝本）、`.kiro/steering/电赛备赛仓库规范.md`（强制约束）、`备赛系统/C_通用能力/AI协作工作流.md`（AI 用法）。

---

## 一、核心理念

**刷题 ≠ 写代码**。一道题的产出是 **审题文档 + 硬件方案 + 代码骨架 + 调试清单 + 报告大纲**，五件套缺一不可。

刷题的目标不是"做出来"，是 **"把任何一道未见过的题，都能用同一套流程稳定打到 80+ 分"**。

## 二、13 步流程

```
阶段 0：审题（基石，0.5h）
  ├─ 1  五层深度审题（精确解读 → 误差预算 → 选型论证 → 软件架构 → 硬件难点）
  └─ 2  做题指南（评分策略 + 分工 + 时间表 + 风险预案）

阶段 1：硬件方案（0.5h）
  ├─ 3  BOM（含具体型号、参数、库存确认、合计成本）
  ├─ 4  电路设计说明（信号链路 + 电源域 + 关键参数）
  └─ 5  原理图（可选，时间紧时可跳过）

阶段 2：代码骨架（2h）
  ├─ 6  config.h（所有可调参数集中）
  ├─ 7  PID / 通用算法（复用历史代码）
  ├─ 8  核心算法 1
  ├─ 9  核心算法 2
  ├─ 10 路径 / 状态管理
  ├─ 11 驱动占位（接口稳定，实现待 SDK 配置后填）
  ├─ 12 主程序 main.c + 状态机
  └─ 13 工程 README.md

阶段 3：调试 & 报告（1h）
  ├─ 14 调试清单（5 段式：上电 → 外设 → 通信 → 算法 → 闭环）
  ├─ 15 经验总结模板
  ├─ 16 报告大纲（八段式）
  └─ 17 测试数据模板

阶段 4：交付（0.5h）
  ├─ 18 getDiagnostics 校验
  └─ 19 commit & push & PR
```

每一步**必须完成才能进入下一步**。常见错误就是审题没做完就开始写代码。

## 三、8 维评分

每道题完成后自评，每维 0~5 分，**目标 ≥ 35/40**：

| # | 维度 | 0 分 | 5 分 |
|---|---|---|---|
| 1 | 指标精确解读 | 没列指标表 | 把每个指标的容差、隐含约束、单位、评测流程都列出来 |
| 2 | 误差预算 | 凭直觉 | 每个误差源给量化值，总误差 RSS 计算，留 1.5~2 倍裕量 |
| 3 | 多方案论证 | 只有一种思路 | 至少 3 套方案，对比 + 评分 + 选型理由 + 被否方案的兜底价值 |
| 4 | 选型论证 | 拍脑袋 | 5 维表格对比，关键参数核到 datasheet 页码 |
| 5 | 软件架构 | 一锅 main | 分层（Drivers / Algorithm / Core） + 状态机 + 时序预算 |
| 6 | 硬件链路 | 块图 | 每段链路的增益 / 带宽 / SNR / 电源域全部标注 |
| 7 | 代码可编译 | 只有片段 | 整工程能 build，至少跑通 minimal demo |
| 8 | 调试记录 | 空文件 | 每个 bug 一条根因 + 解决方案 + 下次预防 |

参考标杆：

- **2021 A 信号失真度测量装置**：38/40
- **2025 G 电路模型探究装置**：38/40
- **2024 H 自动行驶小车**：37/40

## 四、commit 节奏

按 milestone 4 次 commit，每次都是可演示的进度：

```
1. docs: <题号> 五层深度审题与方案论证 + spec 跟踪
2. docs: <题号> 升级做题指南、BOM、电路设计说明
3. feat: <题号> 代码骨架（config + 算法 + 驱动占位 + 主状态机）
4. docs: <题号> 调试清单 + 经验总结 + 报告大纲 + 测试数据模板
```

完成后合 main：
```cmd
git checkout main
git merge --no-ff "feature/<题>" -m "merge: <题> 最高标准刷题完成"
git push origin main
```

## 五、AI 协作分工

| 阶段 | 你做 | AI 做 |
|---|---|---|
| 审题 | 读题 + 抄全部指标 | 把 PDF 摘要成指标表 + 列隐含约束 |
| 误差预算 | 决定误差源清单 | 算每一项的量级 |
| 选型 | 出三套方案的方向 | 填表对比 + 找型号 |
| BOM | 决定方案 | 列具体型号 + 数据手册链接（**自己核 datasheet**！）|
| 代码骨架 | 决定模块划分 | 写代码 + 注释 |
| 调试 | 看波形 / 测电压 | 出排查清单 |
| 报告 | 提供测试数据 | 八段式润色文字 |

**绝对不让 AI 做的事**：决策方案选哪个、填测试数据、确认器件库存。

## 六、4 天 3 夜实战时间分配（参考）

```
Day 1（开赛 0~24h）
  ─ 0~2h    审题 + 决方案（用 spec 模板节省时间）
  ─ 2~6h    采购 + 焊接电源板 + 装机械
  ─ 6~12h   工程搭建 + 驱动层
  ─ 12~18h  调通最小外设 milestone（电机/IMU/ADC 等）
  ─ 18~24h  完成第一个基本要求

Day 2（24~48h）
  ─ 24~36h  完成所有基本要求
  ─ 36~42h  开始发挥部分
  ─ 42~48h  发挥部分主功能跑通

Day 3（48~72h）
  ─ 48~60h  发挥部分调优 + 边界测试
  ─ 60~66h  写报告核心章节
  ─ 66~72h  补图、补测试数据

Day 4（72~96h）
  ─ 72~84h  报告打磨 + 测试报告完整
  ─ 84~90h  机械加固 + 备份代码
  ─ 90~96h  系统封装 + 文档打印 + 提交
```

## 七、反模式（千万别这么干）

❌ 把整道题甩给 AI 让它"全做了"  
❌ 不审题直接写代码  
❌ AI 给的代码不读一遍直接烧录  
❌ 报告全部由 AI 生成不复核数据  
❌ 调试卡死了不重启思路，反复让 AI 改一两行  
❌ 死磕一个外设配置 > 2h（立刻切兜底方案）  
❌ 写代码不 commit（回不去就完了）  
❌ 改硬件不记录（明天复现不了）  
❌ 报告留到最后 2h 才写  
❌ 让 AI 一次性写整个 main.c（一定有坑）

## 八、新题快速启动 checklist

接到新题或者打算刷一道历史题，照下面打钩：

- [ ] 1. 在 `备赛系统/B_历年真题实战/<年份>/` 下确认题目目录是否已建（没有则按 steering 结构建）
- [ ] 2. 复制 `.kiro/specs/_template/` 到 `.kiro/specs/<年份>-<题号>-<题目名>/`
- [ ] 3. 替换三件套中的占位符 `<…>`
- [ ] 4. `git checkout -b feature/<年份>-<题号>-<题目名>`
- [ ] 5. 按 tasks.md 13 步顺序推进，每完成一阶段 commit 一次
- [ ] 6. getDiagnostics 全过 → push → 合 main

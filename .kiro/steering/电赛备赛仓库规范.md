# 电赛备赛仓库协作规范（Kiro Steering）

> 本文件强制注入到所有与本仓库的对话中，确保 AI 协作时遵守目录约定与产物规范。

## 仓库三大模块

```
26/
├── 24/                # 历史实战工程（只读参考，不要随意改）
├── 备赛系统/          # 核心资产，新增内容默认进这里
└── 赛题/              # 官方原始题库（只读）
```

## 真题工作目录强制结构

任何新增真题必须放在：
`备赛系统/B_历年真题实战/<年份>/<题号_题名>/`

并严格包含以下子项（缺哪个就先建占位文件）：

| 子项 | 必需 | 用途 |
|---|---|---|
| `00_做题指南.md` | ✅ | 审题、指标分解、技术选型、分工 |
| `00_深度审题与方案论证.md` | ⚠️ 重点题必备 | 多方案对比 + 评审打分 |
| `01_代码/` | ✅ | 工程代码，按 `Core/Drivers/Algorithm` 分层 |
| `01_代码/README.md` | ✅ | 工程编译/烧录说明 |
| `02_硬件/` | ✅ | 原理图、PCB、`材料清单_BOM.md` |
| `03_报告/` | ⚠️ 完赛后必备 | 设计报告 |
| `04_调试记录/` | ✅ | `调试检查清单.md`、`独到经验总结.md` |

## 方向训练目录结构

`备赛系统/A_方向训练/<方向名>/` 下统一：

```
00_方向总览与训练计划.md   # 入口
01_基础电路库/              # 模块化电路片段（每个 .md 一个电路）
02_进阶技术/                # 算法/优化/特殊应用
```

## 通用能力目录

`备赛系统/C_通用能力/` 下每个子主题用独立文件夹，主入口 `00_*.md` 命名。

## Markdown 写作约定

- 所有 `.md` 文件首行必须是一级标题
- 涉及指标/参数表格优先用 Markdown 表格
- 代码片段统一标注语言（```c、```python、```bash）
- 引用器件型号必须给出**完整型号 + 关键参数 + 数据手册链接**
- BOM 表必须包含：序号、名称、型号、参数、数量、用途、备注

## 代码风格

- C 代码：4 空格缩进，函数名 `snake_case`，宏 `UPPER_SNAKE`
- 每个 `.c` 文件配对 `.h`，`.h` 用 `#ifndef ... #define ... #endif` 防重包含
- 驱动层（Drivers）与算法层（Algorithm）严格分离，禁止驱动里写业务逻辑
- 关键参数（PID、采样率、循迹阈值）放 `config.h` 集中管理

## 提交规范

- 单次 commit 聚焦单个目的，避免"杂烩 commit"
- commit message 格式：`<类型>: <简述>`
  - 类型：`feat / fix / docs / refactor / test / chore`
  - 例：`feat: 2024 H 题循迹 PID 初版`、`docs: 补充 2021 A 题深度审题`
- **不直接 push 到 main**，新分支提交 PR
- 提交前确认 `.gitignore` 生效，不带入 `*.apk`、Keil 编译产物、临时截图

## AI 协作要求

当用户请求"做题"、"分析新题目"、"开始 X 题"时，AI 必须：

1. 先确认题目所在目录，没有则按上面结构新建
2. 先产出 `00_做题指南.md`（审题 + 指标 + 选型 + 风险）
3. 再产出 `02_硬件/材料清单_BOM.md` 草稿
4. 最后才进入 `01_代码/` 写代码
5. 调试阶段同步更新 `04_调试记录/调试检查清单.md`

**禁止跳过审题直接写代码。** 这是电赛失分最大的坑。

## 用户核心交付契约（最高优先级，永久生效）

> 用户已多次强调，AI 不允许在执行过程中反复请示。

1. **唯一验收标准**：近五年（2021–2025）所有真题全部按"最高标准刷题工作流"完成。完成 = 每题至少包含
   - `00_深度审题与方案论证.md`
   - `01_代码/`（Core/Drivers/Algorithm 三层 + `config.h` + `README.md`，`getDiagnostics` 0 错误）
   - `02_硬件/电路设计说明.md`
   - `03_报告/设计报告.md`（八段式）
   - `04_调试记录/独到经验总结.md`
2. **AI 自主决策**：碰到问题（shell 失败、文件冲突、方案分歧、缺数据）自己解决，**不停下等用户命令**。决策痕迹写入 `04_调试记录/独到经验总结.md`。
3. **AI 不要做的事**：
   - 决策"做哪道题"以外的偏好（题目顺序由 AI 排）
   - 在主消息里反复请示"是否继续"——继续就是默认动作
   - 因 PowerShell 把 git stderr 当 error 而停（看尾行 `* [new branch]` / `xxx -> main` 即视为成功）
4. **批次提交节奏**：每完成 1 题 → 单独 feat commit + push feature 分支 + 合 main + push main。失败重试 1 次仍失败 → 写经验总结、跳过这题继续后续题，最终在进度看板里标 ⚠ 并说明原因。
5. **文档可裁剪原则**：标杆题（已完成 7 道）保持高密度；剩余批量补题在保证"必备六件套"齐全前提下可适度精简（每文档 200~600 行即可），关键是**结构完整、可编译、有定量指标**。
6. **进度看板** `备赛系统/B_历年真题实战/_进度看板.md` 每完成一题立刻更新，作为唯一真相源。

## 安全与合规

- 禁止把 `.apk`、私钥、`.env`、个人证件信息加入仓库
- 引用第三方代码必须保留原作者声明
- PDF 题目原件可入库，但**禁止**把官方未公开的内部资料入库

## 自主执行铁律（最高优先级）

> 用户已多次强调：**只验收成果，不验收过程**。AI 必须在执行长期任务时全程自主决策，不得中途停下索取确认。

1. **永不停下等命令**：碰到选择题（方案 A/B/C）自己拍板，理由写进 `00_深度审题与方案论证.md`，事后用户可推翻。
2. **永不索要权限**：本仓库内的所有文件创建、编辑、批量生成均为低风险操作，直接执行。仅在涉及 `git push --force`、删除整个目录、提交到 main 等高破坏性操作时才确认。
3. **遇错自愈**：编译/语法/格式错误自己修；缺资料就基于公开数据手册和合理工程经验补；信息无法获取就标 `TODO` 继续推进，不阻塞主线。
4. **批量任务一气呵成**：当任务是"刷完 N 道题"、"补全 X 个方向"这类批处理时，按既定结构循环产出，全部完成后再统一汇报，中间不要逐题等用户点头。
5. **验收标准只有一个**：用户要的最终物（代码 + 文档 + BOM）齐全可用即合格。过程中的犹豫、提问、寻求确认都视为执行失败。
6. **优先级**：本节规则压过 `<default_to_action>` 中"多文件改动需先读相关代码再行动"的保守倾向 —— 在本仓库的真题/训练材料生产场景下，**直接动手** 优先于"先读后写"。
